> 版本：v1.0  
> 日期：2026-04-13  
> 状态：待评审  
> 需求来源：[Elo-Mario/openclaw-memory-system](https://github.com/Elo-Mario/openclaw-memory-system) 对标分析 + 团队实际痛点

---

## 1. 背景与目标

### 1.1 背景

我们的 agent-memory 服务已具备核心能力：语义向量搜索、多 Agent 隔离、实时自动捕获（autoCapture）、自动分类、TTL 生命周期管理。但在实际使用中暴露出以下问题：

1. **记忆碎片化**：autoCapture 实时提取单条内容，缺少跨会话的回顾提炼，导致高层模式（如"这个 Agent 反复遇到同类问题"）难以被发现
2. **经验不沉淀**：任务完成后，踩坑经验和可复用流程散落在各条记忆中，没有系统化整理
3. **归档粗放**：TTL 仅按创建时间过期，未考虑访问热度——一条高优先级但很少被检索的知识可能仍有价值

### 1.2 目标

在不破坏现有架构的前提下，新增三个能力：

| 能力 | 目标 | 优先级 |
|------|------|--------|
| Dream 回顾提炼 | 定期扫描记忆，识别跨会话模式，生成高层洞察 | P0 |
| 任务复盘模板 | 任务完成时自动触发复盘，结构化沉淀经验 | P1 |
| 热度感知归档 | 基于访问热度智能归档，替代纯时间过期 | P2 |

### 1.3 约束

- 所有新能力以 API 端点形式暴露，由 cron 或 OpenClaw 插件调用
- 不修改现有 API 的行为和接口
- 不引入新的外部依赖（复用现有 Qdrant + SQLite）
- 保持 Go 单二进制部署，不依赖外部 LLM API 做提炼（可选增强）

## 2. 需求详述

### 2.1 Dream 回顾提炼（P0）

#### 2.1.1 用户故事

作为团队管理者，我希望系统每天凌晨自动回顾所有 Agent 的近期记忆，识别重复出现的模式、积累的经验和潜在的问题，生成结构化的洞察记忆，这样我不需要手动翻阅每个 Agent 的记忆日志。

#### 2.1.2 功能需求

**FR-DREAM-001：Dream API 端点**
- 新增 `POST /api/v1/dream` 端点
- 参数：`{ "agent_id": "all" | "<specific>", "lookback_days": 1, "dry_run": false }`
- 支持指定单个 Agent 或全部 Agent
- 支持 dry_run 模式：只分析不写入，返回将生成的洞察

**FR-DREAM-002：模式识别**
- 扫描指定时间窗口内的所有记忆
- 识别以下模式：
  - **重复模式**：多条记忆内容相似度 > 阈值（如 0.7），说明某类问题反复出现
  - **趋势模式**：同一 category 的记忆数量在时间窗口内显著增加
  - **孤立记忆**：高优先级但从未被访问过的记忆
  - **冲突模式**：同一 topic 的记忆内容存在矛盾（如"应该做 X" vs "不应该做 X"）

**FR-DREAM-003：洞察生成**
- 对每个识别到的模式，生成一条洞察记忆
- 洞察记忆的 category 为 `knowledge`，priority 为 2
- 洞察记忆的内容格式：
  ```
  [Dream 洞察] <模式类型>: <简要描述>
  
  来源记忆: <相关记忆ID列表>
  时间窗口: <起止日期>
  模式描述: <详细说明>
  建议行动: <可选>
  ```
- 洞察记忆的 tags 包含 `dream`、`<模式类型>`、日期

**FR-DREAM-004：去重与合并**
- 如果新生成的洞察与已有洞察相似度 > 0.85，不重复创建，而是更新已有洞察（追加新的来源记忆）
- 每条洞察最多关联 20 条来源记忆，超出时压缩最早的来源

**FR-DREAM-005：Dream 日志**
- 每次 Dream 执行后，生成一条 Dream 日志记忆（category: `working`, tags: `dream-log`）
- 内容包含：扫描记忆数、发现模式数、新增/更新洞察数、耗时
- 便于追踪 Dream 效果

#### 2.1.3 非功能需求

- NFR-DREAM-001：单次 Dream 执行（6个Agent，7天回溯）应在 30 秒内完成
- NFR-DREAM-002：Dream 期间不影响正常读写 API 的性能
- NFR-DREAM-003：支持 cron 调用，返回标准 JSON 响应

### 2.2 任务复盘模板（P1）

#### 2.2.1 用户故事

作为团队管理者，我希望当一个 Agent 完成任务后，系统能自动分析任务过程中的记忆变化，按照结构化模板生成复盘总结，这样团队经验可以系统化积累。

#### 2.2.2 功能需求

**FR-REVIEW-001：复盘 API 端点**
- 新增 `POST /api/v1/review` 端点
- 参数：`{ "agent_id": "<agent>", "since": "<ISO时间>" }`
- 扫描指定时间窗口内该 Agent 的所有记忆变化

**FR-REVIEW-002：五问分析**
- 基于扫描到的记忆，按照"沉淀五问"框架进行分析：
  1. **踩坑了吗？** → 识别标记为 working 且包含错误/失败/问题的记忆
  2. **可复用流程？** → 识别重复出现的操作步骤（≥2次相似）
  3. **抽象准则？** → 识别 principle 类记忆中可能需要补充的规则
  4. **跨场景泛化？** → 识别适用于多个 Agent 的 knowledge
  5. **问答值得留存？** → 识别包含具体问题+答案的记忆

**FR-REVIEW-003：复盘报告生成**
- 输出结构化 JSON 报告，包含：
  ```json
  {
    "agent_id": "m10s",
    "time_range": { "from": "...", "to": "..." },
    "summary": { "total_memories": 50, "categories": {...} },
    "findings": {
      "experience": ["发现1", "发现2"],
      "skills": ["流程1"],
      "principles": ["准则1"],
      "insights": ["洞察1"],
      "queries": ["问答1"]
    },
    "action_items": ["建议行动1", "建议行动2"],
    "related_memories": ["id1", "id2", ...]
  }
  ```

**FR-REVIEW-004：OpenClaw 插件集成**
- 插件新增 `memory_review` 工具，供 Agent 主动调用
- 插件在 autoCapture 后检测任务结束信号（如用户说"完成了"/"搞定了"），自动触发复盘

#### 2.2.3 非功能需求

- NFR-REVIEW-001：单次复盘（7天窗口）应在 10 秒内完成
- NFR-REVIEW-002：复盘报告不自动写入记忆，由 Agent/用户决定是否存储

### 2.3 热度感知归档（P2）

#### 2.3.1 用户故事

作为系统管理员，我希望归档策略同时考虑时间和访问热度，避免有价值但访问频率低的记忆被过早清理，也避免高访问但已过时的记忆占据资源。

#### 2.3.2 功能需求

**FR-HEAT-001：热度评分算法**
- 基于现有 memory 模型的 `access_count` 和 `last_accessed` 字段计算热度分
- 公式：
  ```
  heat_score = access_count * recency_weight + time_since_last_access * freshness_weight
  ```
  其中 recency_weight 和 freshness_weight 通过 config.yaml 配置
- 热度分范围 0-100

**FR-HEAT-002：增强 TTL 策略**
- 在现有 TTL 基础上增加热度维度：
  - `active` → `degraded`：当 TTL 到期 **且** heat_score < 阈值时触发
  - 如果 heat_score >= 阈值（高优先级+有访问），延长 TTL 一个周期
- 新增配置项：
  ```yaml
  ttl:
    heat_threshold: 30        # 热度低于此值才降级
    heat_extension_multiplier: 1.5  # 高热度记忆 TTL 延长倍数
  ```

**FR-HEAT-003：热度报告**
- 在现有 `/api/v1/memories/report` 健康报告中增加热度分布：
  - 热门记忆 Top 10（heat_score 最高）
  - 冷门记忆 Top 10（高优先级但 heat_score 最低）
  - 热度分布直方图数据

#### 2.3.3 非功能需求

- NFR-HEAT-001：热度计算不增加正常读写延迟
- NFR-HEAT-002：与现有 TTL 扫描复用同一调度周期（6h）

## 3. 验收标准

| 编号 | 验收条件 |
|------|----------|
| AC-001 | `POST /api/v1/dream` 能成功执行并返回模式分析结果 |

| AC-008 | 新增端点不影响现有 API 性能（P99 < 500ms） |
